A tour of DMD

How a chunk of D source text becomes machine code — and the data structures it morphs through on the way. Each panel in the explorer is a peek at one of these.

source text→ tokens & AST→ typed AST→ backend IR→ machine code

The compiler is a pipeline. Text comes in the left; an object file falls out the right. In between, your program is repeatedly rewritten into ever-lower-level shapes. Every link below points at the actual line in dlang/dmd where that shape is defined — go read it, it's friendlier than it looks.

1 · Parsing — text becomes a tree

No meaning is attached yet; this stage is pure shape. The lexer turns bytes into tokens, and the parser assembles tokens into a tree mirroring the grammar.

The lexerTokens pane

The lexer scans raw bytes and emits a flat stream of tokens. Its real work is recognising the kinds of lexeme: a hex integer, a floating-point literal, a string with an escape — each collapses to a single token carrying its decoded value. Try mangling a number or leaving a string unterminated and watch the stream react.

▶ Tokens — one line per source line. The literals 0xDEAD_BEEF, 3.14e2 and "a\tb" each become a single TOK with a value attached; keywords and operators get their own.

Tokens — enum TOK

One value per lexeme kind: int32, float64, string, plus… The Tokens pane lists these per source line.

tokens.d lexer.d

The parserParse tree pane

Recursive descent: the parser reads tokens left to right and builds an abstract syntax tree whose nesting encodes precedence. The flat a + b * 2 becomes a tree where the multiply sits below the add — so it binds tighter — without any parentheses in the source.

▶ Parse tree — the AST straight from the parser, before any checking. Note how a + b * 2 nests a MulExp inside an AddExp: the tree, not the text, carries the precedence.

The parser

A hand-written recursive-descent parser. Reads tokens, builds AST nodes. The Parse tree pane is its raw output, before any checking.

parse.d

Expressions — class Expression

Base class for everything that has a value: AddExp, MulExp, VarExp, literals… Each subclass is a node kind.

expression.d

Statements & symbols

Statement covers control flow (IfStatement, loops); Dsymbol is the base for named things — modules, functions, variables, aggregates.

statement.d dsymbol.d

Printing the tree — printast.d

Both the Parse tree and Semantic tree panes are rendered here. It walks the AST and prints each node by its internal class name — so you see AddExp, not a + b.

printast.d

2 · Semantic — the tree gains meaning

DMD walks the AST in several passes and works out what everything means: it resolves names to symbols, assigns a type to every expression, runs static if/templates/CTFE, and checks attributes.

Before & after typingParse + Semantic panes

The parser leaves an untyped tree; semantic analysis resolves every name to a symbol and pins a type on every expression — inserting the implicit conversions the source left unwritten. The two right-hand panes are the same function: the Parse tree straight from the parser, and the Semantic tree after analysis. Returning an int sum as a long makes a CastExp appear out of nowhere.

▶ Parse tree vs Semantic tree for the same source. After analysis the return expression is wrapped in a CastExp to long — the implicit int→long conversion the parser had no way to know about.

Name lookup — Scope

Analysis threads a Scope chain through the tree; each identifier is resolved against it to the Dsymbol it names — a parameter, a local, a function.

dscope.d

Implicit conversions — implicitCastTo

When an expression's type doesn't match where it's used, the analyzer inserts a CastExp — the int→long you can watch appear between the two panes.

dcast.d

LoweringLowered AST pane

Beyond type-checking, semantic analysis lowers high-level constructs into simpler ones the back end already knows how to emit. A foreach over a range becomes a plain for; an append ~= becomes a call into druntime. The Lowered AST pane (-vcg-ast) regenerates the tree as D after this rewriting, so you can read the desugared form directly.

▶ Lowered AST (-vcg-ast) — the tree after semantic analysis, regenerated as D. Watch the rewrite: the foreach becomes a plain for, and r ~= … a runtime call (_d_arrayappendcTX).

The passes

Roughly: symbols (dsymbolSemantic), then types (typeSemantic), then expressions/bodies (expressionSemantic, semantic3).

dsymbolsem.d typesem.d expressionsem.d

Types — class Type

Every typed expression points at a Type: TypeBasic (int), TypePointer, TypeFunction, TypeStruct…

mtype.d

Functions & aggregates

FuncDeclaration holds a function's body, signature and analysis flags; StructDeclaration/ClassDeclaration hold fields & methods.

func.d dstruct.d dclass.d

The module — class Module

The root of it all: one source file, its members, and the entry point semantic analysis starts from.

dmodule.d
🧠 Front end vs. back end: everything so far is the front end and is shared by every D compiler (DMD, GDC, LDC). What follows is DMD's own back end — the part that actually emits x86.

3 · Codegen — the back end emits machine code

The glue layer translates each function's typed AST into the back end's own IR — a graph of basic blocks, each holding elem expression trees — which the optimizer rewrites and code generation turns into x86.

The back-end IRBackend IR pane

The back end has its own, far leaner representation than the front-end AST: no types-as-objects, just an operator and operands. Here the -O optimizer earns its keep — constant folding evaluates 2 + 3 at compile time, so the runtime multiply is against a literal 5. (Open it in the full explorer to also see the pre-optimizer IR and diff them.)

▶ Backend IR — each function's block/elem graph after the -O optimizer. Nodes carry backend enums (OPmul, OPvar, TYint); the optimizer has already folded 2 + 3 to a constant.

Expression nodes — struct elem

The back end's expression tree. E1/E2 are the child operands. Far leaner than the front-end AST — no types-as-objects, just an operator and operands.

backend/el.d

Operators — enum OPER

What an elem does: OPadd, OPmul, OPvar, OPconst, OPcall… These are the node labels in the IR pane.

backend/oper.d

Back-end types — enum TY

The back end's own flat type tags — TYint, TYdouble, TYnptr — carried on every elem. No relation to the front end's Type classes.

backend/ty.d

Basic blocks — struct block / enum BC

A function is a graph of blocks chained by Bnext and linked by successor edges. BC is how a block exits: BC.goto_, BC.iftrue, BC.retexp…

backend/cc.d

The optimizer — optfunc

Runs only with -O: constant folding, common-subexpression elimination, dead-code removal, loop rotation… It rewrites the block/elem graph in place — that's the gap between the pre- and post-opt IR.

backend/go.d backend/blockopt.d

Code generationDisassembly pane

The final pass walks the optimized blocks and selects instructions, mapping each elem to x86. It's smarter than a literal translation: a multiply by a small constant is strength-reduced into an address computation, so x * 3 emerges as a single lea rather than an imul.

▶ Disassembly (-vasm) — the actual x86-64 instructions for that IR. Watch x * 3 become a single lea (address arithmetic) instead of a multiply. Edit the source and the machine code follows.

Code gen & disassembly

The final pass walks the optimized blocks and selects x86 instructions; the disassembler renders them for the -vasm (Disassembly) pane.

x86/cgcod.d x86/disasm86.d
