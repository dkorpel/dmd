A tour of DMD 🧭

How a chunk of D source text becomes machine code — and the data structures it morphs through on the way. Each panel in the explorer is a peek at one of these.

source text→ tokens & AST→ typed AST→ backend IR→ machine code

The compiler is a pipeline. Text comes in the left; an object file falls out the right. In between, your program is repeatedly rewritten into ever-lower-level shapes. Every link below points at the actual line in dlang/dmd where that shape is defined — go read it, it's friendlier than it looks.

1 · Parsing — text becomes a tree

First the lexer chops the raw bytes into tokens (keywords, numbers, punctuation). Then the parser assembles those tokens into an abstract syntax tree (AST): a literal tree of objects mirroring the grammar — a function holds statements, a statement holds expressions, an a + b becomes an AddExp with two children. No meaning is attached yet; it's pure shape.

Tokens — enum TOK

One value per lexeme kind: int32, identifier, plus… The Tokens pane lists these per source line.

tokens.d lexer.d

The parser

A hand-written recursive-descent parser. Reads tokens, builds AST nodes. The Parse tree pane is its raw output, before any checking.

parse.d

Expressions — class Expression

Base class for everything that has a value: AddExp, CallExp, VarExp, literals… Each subclass is a node kind.

expression.d

Statements & symbols

Statement covers control flow (IfStatement, loops); Dsymbol is the base for named things — modules, functions, variables, aggregates.

statement.d dsymbol.d
🔎 The Parse tree and Semantic tree panes are both rendered by printast.d, which walks the AST and prints each node by its internal class name — so you see AddExp, not a + b.

2 · Semantic — the tree gains meaning

Now DMD walks the AST in several passes and figures out what everything means: it resolves names to symbols, assigns a type to every expression, runs static if/templates/CTFE, checks @safe/ nothrow, and lowers high-level constructs into simpler ones (a foreach becomes a plain for, ~= becomes a runtime call). The Lowered AST pane (-vcg-ast) shows the tree after this rewriting.

The passes

Roughly: symbols (dsymbolSemantic), then types (typeSemantic), then expressions/bodies (expressionSemantic, semantic3).

dsymbolsem.d typesem.d expressionsem.d

Types — class Type

Every typed expression points at a Type: TypeBasic (int), TypePointer, TypeFunction, TypeStruct…

mtype.d

Functions & aggregates

FuncDeclaration holds a function's body, signature and analysis flags; StructDeclaration/ClassDeclaration hold fields & methods.

func.d dstruct.d dclass.d

The module — class Module

The root of it all: one source file, its members, and the entry point semantic analysis starts from.

dmodule.d
🧠 Front end vs. back end: everything so far is the front end and is shared by every D compiler (DMD, GDC, LDC). What follows is DMD's own back end — the part that actually emits x86.

3 · Codegen — the back end emits machine code

The glue layer translates each function's typed AST into the back end's own IR: a graph of basic blocks, each holding a forest of elem expression trees. The optimizer rewrites that graph; then code generation turns it into x86 instructions, which the disassembler prints. The explorer shows the IR before and after the optimizer plus the final disassembly — diff the two IR panes to watch the optimizer work.

Expression nodes — struct elem

The back end's expression tree. E1/E2 are the child operands. Far leaner than the front-end AST — no types-as-objects, just an operator and operands.

backend/el.d

Operators — enum OPER

What an elem does: OPadd, OPmul, OPvar, OPconst, OPcall… These are the node labels in the IR panes.

backend/oper.d

Back-end types — enum TY

The back end's own flat type tags — TYint, TYdouble, TYnptr — carried on every elem. No relation to the front end's Type classes.

backend/ty.d

Basic blocks — struct block / enum BC

A function is a graph of blocks chained by Bnext and linked by successor edges. BC is how a block exits: BC.goto_, BC.iftrue, BC.retexp…

backend/cc.d

The optimizer — optfunc

Runs only with -O: constant folding, common-subexpression elimination, dead-code removal, loop rotation… It rewrites the block/elem graph in place — that's the gap between the two IR panes.

backend/go.d backend/blockopt.d

Code gen & disassembly

The final pass walks the optimized blocks and selects x86 instructions; the disassembler renders them for the -vasm (Disassembly) pane.

x86/cgcod.d x86/disasm86.d
⚙️ The two IR panes are dumped by this playground itself — a small printer in compiler/wasm/dmdwasm.d walks the same block/ elem graph and labels each node with its real OPER/TY/BC name.
